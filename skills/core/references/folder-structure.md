# Canonical Target Repository Structure

This document defines the structure that `init_repo.py` creates when scaffolding a mycelium-enabled living repository. Every directory and file serves a specific purpose in the self-documenting framework.

## Full Structure

```
project-root/
├── CLAUDE.md                              # AI agent instructions (generated from template)
├── ENVIRONMENTS_INSTALLATIONS.md          # All env setup, dependencies, gotchas
├── .living/                               # The memory layer
│   ├── decisions.md                       # Append-only log: why choices were made
│   ├── learnings.md                       # Append-only log: gotchas, surprises, insights
│   ├── conventions.md                     # Repo-specific overrides to defaults
│   ├── conventions/                       # Installed convention packs
│   │   └── ACTIVE_CONVENTIONS.yaml
│   ├── generated-conventions/             # Auto-generated from crystallize mode
│   ├── log/                               # Session-by-session work logs
│   │   ├── LOG_REGISTRY.md                # Scannable index of all sessions
│   │   └── YYYY-MM-DD-NNN-<slug>.md       # One file per session
│   └── findings/
│       ├── FINDINGS_REGISTRY.md
│       └── {topic-slug}.md
├── algorithms/
│   ├── ALGORITHM_MANIFEST.md              # Registry with YAML entries
│   └── [algorithm-name]/
│       ├── ALGORITHM_NAME.md              # UPPER_SNAKE_CASE of folder name
│       └── ...
├── analysis/
│   ├── ANALYSIS_MANIFEST.md
│   └── [analysis-name]/
│       ├── ANALYSIS_NAME.md               # UPPER_SNAKE_CASE of folder name
│       ├── scripts/                       # Marimo notebooks and/or Python scripts
│       ├── outputs/                       # Figures, tables, intermediate results
│       └── reports/                       # LaTeX writeups
├── data/
│   ├── DATA_MANIFEST.md
│   ├── raw/                               # Immutable originals
│   ├── processed/                         # Cleaned/transformed
│   └── metadata/                          # Schemas, data dictionaries, provenance
├── reference_material/
│   ├── REFERENCE_MANIFEST.md
│   └── ...
└── todo/
    ├── TODO_REGISTRY.md                    # Master registry of all todo items
    ├── TODO_ITEM_TEMPLATE.md               # Template for new items
    └── [item-name].md                      # Detailed writeup per item
```

## Naming Conventions

### Manifest files

Each top-level directory has a descriptive manifest: `ANALYSIS_MANIFEST.md`, `DATA_MANIFEST.md`, `ALGORITHM_MANIFEST.md`, `REFERENCE_MANIFEST.md`. The prefix makes manifests instantly identifiable in search results.

### Subdirectory documentation

Every subdirectory (analysis, algorithm, dataset) has a documentation file named after the folder in UPPER_SNAKE_CASE. For example:
- `analysis/snp-analysis/` → `SNP_ANALYSIS.md`
- `algorithms/deseq2-wrapper/` → `DESEQ2_WRAPPER.md`
- `data/raw/patient-cohort-2024/` → `PATIENT_COHORT_2024.md`

This makes documents discoverable in search — a list of 20 `README.md` files tells you nothing, while `SNP_ANALYSIS.md`, `DOSE_RESPONSE.md`, `PATHWAY_ENRICHMENT.md` are self-documenting.

**Conversion rule**: Replace hyphens with underscores, uppercase everything. `my-analysis-name` → `MY_ANALYSIS_NAME.md`.

## Directory Purposes

### `.living/` — The Memory Layer

This is the heart of a mycelium-enabled repo. It stores the accumulated intelligence that makes the repo "living."

- **`decisions.md`**: Append-only log documenting *why* choices were made. Every non-obvious decision (data exclusions, algorithm selections, parameter choices) gets an entry. This prevents repeating past reasoning.
- **`learnings.md`**: Append-only log of gotchas, surprises, and insights. When something unexpected happens — an edge case, a library quirk, a failed approach — record it here so the next session benefits.
- **`conventions.md`**: Repo-specific overrides to mycelium defaults or domain skill conventions. If your project handles things differently, document it here rather than fighting the framework.
- **`conventions/`**: Installed convention packs. `ACTIVE_CONVENTIONS.yaml` lists what's installed and where to find conventions.
- **`generated-conventions/`**: Conventions auto-generated by the `crystallize` mode from accumulated learnings. These are candidates for contribution back to the network.
- **`log/`**: Session-by-session work logs. Each session creates a dated markdown file (`YYYY-MM-DD-NNN-<slug>.md`) with timestamped action entries. Logs are built incrementally throughout the session via hook-injected directives and finalized at session end.
- **`log/LOG_REGISTRY.md`**: Scannable index of all sessions — date, project, duration, summary, outputs, and status.
- **`findings/`**: Scientific findings organized by topic, with evidence ledgers.
- **`findings/FINDINGS_REGISTRY.md`**: Per-project table-of-contents — one row per finding with ID, claim, status, topic link, implications, tags, and last updated date. Created when the first finding is crystallized; regenerated by `crystallize_findings.py`.
- **`findings/{topic-slug}.md`**: One file per topic; findings accumulate evidence across runs.

### `algorithms/` — Reusable Computational Methods

Algorithms that could be used across multiple analyses. Each gets its own subdirectory with a documentation file (UPPER_SNAKE_CASE of the folder name, e.g., `DESEQ2_WRAPPER.md`) documenting its purpose, inputs, outputs, and parameters.

### `analysis/` — Analytical Work

Each analysis gets its own subdirectory with a standardized structure. Analyses can reference parent analyses to form lineage chains. The `ANALYSIS_MANIFEST.md` provides a registry of all analyses with their status and relationships.

### `data/` — Data Assets

- **`raw/`**: **Immutable.** Original data files are placed here and never modified. This is the single source of truth. If raw data needs correction, add a note to the manifest and create a processed version.
- **`processed/`**: Cleaned, transformed, or derived datasets. Each should document its provenance (which raw data, what transformations).
- **`metadata/`**: Data dictionaries, schemas, provenance documents, and summary statistics. Every dataset in `raw/` or `processed/` should have a corresponding metadata entry.

### `reference_material/` — External References

Papers, documentation, external datasets used for comparison, or any reference material relevant to the project.

### `todo/` — Future Work Items

A structured tracker for future work, ideas, and planned improvements. `TODO_REGISTRY.md` is the master registry — a table of all items with their priority, status, category, date, and author. Each item has its own `[item-name].md` file with a full writeup (description, motivation, proposed approach, acceptance criteria). `TODO_ITEM_TEMPLATE.md` provides the template for new items.

Priority levels: `critical`, `high`, `medium`, `low`, `idea`. Status values: `open`, `in-progress`, `blocked`, `complete`, `wont-do`. Use the mycelium `todo-idea` mode (or `/todo-idea`) to add new items interactively.

## Manifest Format

Each top-level directory contains a descriptive manifest file (`ANALYSIS_MANIFEST.md`, `DATA_MANIFEST.md`, etc.) that serves as a registry. Entries use YAML blocks with prose annotations.

### Example: Data Manifest Entry

```yaml
# data/DATA_MANIFEST.md

## Datasets

### patient-cohort-2024
```yaml
name: patient-cohort-2024
type: clinical
source: Hospital X de-identified export
date_acquired: 2024-03-15
format: CSV (3 files)
rows: 12847
columns: 42
raw_path: data/raw/patient-cohort-2024/
metadata_path: data/metadata/patient-cohort-2024/
status: validated
known_issues:
  - 3 patients have missing age data
  - Dates prior to 2020 use a different coding scheme
tags: [clinical, longitudinal, demographics]
```

Cohort data covering 12,847 patients over a 4-year period. Includes demographics, treatment history, and outcome measures. Missing age data affects patients P-1042, P-3891, and P-7234 — these are excluded from age-stratified analyses but included in all others.
```

### Example: Analysis Manifest Entry

```yaml
# analysis/ANALYSIS_MANIFEST.md

## Analyses

### differential-expression-treatment-v1
```yaml
name: differential-expression-treatment-v1
status: complete
created: 2024-04-01
last_updated: 2024-04-18
datasets: [patient-cohort-2024, rnaseq-batch-1]
algorithms: [deseq2-wrapper]
parent_analysis: null
key_findings:
  - 342 genes differentially expressed (FDR < 0.05)
  - Strong enrichment in inflammatory pathways
report: analysis/differential-expression-treatment-v1/reports/de-report.tex
tags: [differential-expression, treatment-response, primary]
```

Initial differential expression analysis comparing treatment responders vs non-responders. Used DESeq2 with default parameters. The inflammatory pathway enrichment was unexpected and warrants follow-up with pathway-specific analysis.
```

### Example: Algorithm Manifest Entry

```yaml
# algorithms/ALGORITHM_MANIFEST.md

## Algorithms

### deseq2-wrapper
```yaml
name: deseq2-wrapper
purpose: Wrapper around DESeq2 for standardized differential expression analysis
language: R
dependencies: [DESeq2, BiocParallel]
inputs: count matrix (genes x samples), sample metadata
outputs: results table (log2FC, pvalue, padj), MA plot, volcano plot
parameters:
  - alpha: significance threshold (default 0.05)
  - min_count: minimum count filter (default 10)
author: project-team
version: 1.0.0
tags: [differential-expression, rnaseq, statistics]
```

Standardized DESeq2 wrapper that enforces consistent preprocessing, filtering, and output formatting across all differential expression analyses in this project.
```

## Key Principles

1. **Raw data is sacred.** Never modify files in `data/raw/`. If something needs fixing, document the issue and create a corrected version in `data/processed/`.

2. **Everything is registered.** If it exists in the repo, it has a manifest entry (in the appropriate `*_MANIFEST.md`). No orphan analyses, no mystery datasets.

3. **Decisions are documented.** The `.living/decisions.md` log ensures that future sessions understand not just *what* was done, but *why*.

4. **Learnings accumulate.** Every gotcha, edge case, and insight goes into `.living/learnings.md`. The repo gets smarter over time.

5. **Reproducibility is non-negotiable.** Every analysis has a `run.sh` or `run.py` that reproduces its outputs from raw data.
